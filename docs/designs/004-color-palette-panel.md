# DES-004: Color Palette Panel

## Overview

A new XRay panel that allows users to create, manage, and generate color palettes with AI assistance, visualization tools, and page integration.

**Goal**: Enable designers to build color palettes through manual picking, algorithmic generation, or AI assistance, and apply them to pages via MCP tools.

**Dependencies**:
- [DES-005: Multi-Panel Architecture](./005-multi-panel-architecture.md) - Required for panel registration and state management

---

## Summary

A new XRay panel that allows users to create, manage, and generate color palettes. Colors can be:
- Created manually by picking colors
- Generated using color theory algorithms (complementary, analogous, etc.)
- Generated by AI through natural language requests
- Applied to the page as CSS variables via MCP tools

## Motivation

When designing or modifying websites, color selection is crucial. Users need a way to:
- Build cohesive color palettes
- Lock certain colors while regenerating others
- Quickly copy color values
- Visualize colors together before applying them
- Apply palettes to pages and see the results instantly
- Export palettes with semantic roles for stakeholder review

### Key Design Decisions

1. **Color Strips vs Squares**: Switched from square swatches to vertical color strips for better visual impact and more display area for controls.

2. **Hover-Reveal Controls**: Controls are hidden by default and appear on hover to keep the UI clean while colors are prominently displayed.

3. **No Background Overlay**: Instead of a dark semi-transparent background behind controls, icons use contrasting colors (light on dark colors, dark on light colors) for a cleaner look.

4. **Horizontal Scrolling**: When too many colors are added, the strip container scrolls horizontally rather than wrapping.

### Data Viz Slide Algorithm

The Data Viz slides (Pie Chart, Bar Chart, Treemap) use a **relationship-aware algorithm** that considers not just individual color properties, but how colors relate to each other perceptually and in terms of contrast.

#### Core Metrics

We compute three inputs for the optimization:

| Metric | Type | Description |
|:-------|:-----|:------------|
| **Perceptual Distance Matrix** | N×N | CIEDE2000 distance between all color pairs (0-100+) |
| **Contrast Matrix** | N×N | WCAG contrast ratio between all pairs (1-21) |
| **Prominence Weights** | N×1 | Individual color visual weight (0-1) |

#### 1. Perceptual Distance Matrix

Uses **chroma.js** `deltaE()` function to calculate CIEDE2000 color differences:

```typescript
// distanceMatrix[i][j] = how different color i looks from color j
function buildDistanceMatrix(colors: PaletteColor[]): number[][] {
  const n = colors.length;
  const matrix: number[][] = Array(n).fill(null).map(() => Array(n).fill(0));

  for (let i = 0; i < n; i++) {
    for (let j = i + 1; j < n; j++) {
      // chroma.deltaE uses CIEDE2000 by default
      const dist = chroma.deltaE(colors[i].hex, colors[j].hex);
      matrix[i][j] = dist;
      matrix[j][i] = dist;  // symmetric
    }
  }
  return matrix;
}
```

**About chroma.deltaE:**
- `chroma.deltaE(a, b)` → CIEDE2000 (default, most perceptually accurate)
- `chroma.deltaE(a, b, 'CIE76')` → Original 1976 Euclidean formula
- `chroma.deltaE(a, b, 'CIE94')` → 1994 improved formula

We use **CIEDE2000** because it accounts for human perception quirks:
- We're more sensitive to certain hue shifts (especially in blue/purple range)
- Lightness differences matter more than chroma in dark colors
- Corrects for the "blue turns purple" issue in earlier formulas

**Interpretation**: deltaE < 1 = imperceptible, < 5 = subtle, > 10 = obvious, > 50 = very different

#### 2. Contrast Matrix

Uses **chroma.js** `contrast()` function to calculate WCAG 2.0 contrast ratios:

```typescript
// contrastMatrix[i][j] = contrast ratio between color i and j
function buildContrastMatrix(colors: PaletteColor[]): number[][] {
  const n = colors.length;
  const matrix: number[][] = Array(n).fill(null).map(() => Array(n).fill(0));

  for (let i = 0; i < n; i++) {
    for (let j = i + 1; j < n; j++) {
      // chroma.contrast implements WCAG 2.0 luminance-based formula
      const ratio = chroma.contrast(colors[i].hex, colors[j].hex);
      matrix[i][j] = ratio;
      matrix[j][i] = ratio;  // symmetric
    }
  }
  return matrix;
}
```

**About chroma.contrast:**
- Implements WCAG 2.0 relative luminance formula: `(L1 + 0.05) / (L2 + 0.05)`
- Returns ratio from 1:1 (identical) to 21:1 (black/white)
- Used for both accessibility compliance and visual separation optimization

**Interpretation**: 1 = identical, 3 = minimum for large text (AA-large), 4.5 = AA normal text, 7 = AAA, 21 = max (black/white)

#### 3. Prominence Weights

Individual color "importance" based on visual properties, calculated using **chroma.js**:

```typescript
function calculateProminence(color: PaletteColor): number {
  const sat = chroma(color.hex).get('hsl.s');           // 0-1 saturation from HSL
  const lum = chroma(color.hex).luminance();            // 0-1 relative luminance (WCAG)
  const lumDist = Math.abs(lum - 0.5) * 2;              // 0-1 (distance from middle gray)
  const contrast = chroma.contrast(color.hex, '#ffffff');
  const normContrast = (contrast - 1) / 20;             // 0-1 normalized

  return Math.max(
    (sat * 0.4) + (lumDist * 0.3) + (normContrast * 0.3),
    0.1  // minimum floor prevents any color from being invisible
  );
}
```

**chroma.js methods used:**
- `chroma(hex).get('hsl.s')` → HSL saturation component (0-1)
- `chroma(hex).luminance()` → WCAG relative luminance (0-1)
- `chroma.contrast(a, b)` → WCAG contrast ratio (1-21)

**Weight rationale:**
- **Saturation (40%)**: Vivid colors command more visual attention
- **Luminance distance (30%)**: Colors far from middle gray (very light/dark) stand out more
- **Contrast vs white (30%)**: High contrast colors are more visually impactful

#### Spatial Representation: Slots

Each visualization type defines **slots** with adjacency relationships:

```typescript
interface Slot {
  id: number;
  importance: number;      // visual prominence of this position (0-1)
  neighbors: number[];     // adjacent slot IDs
  weight: number;          // relative size (for pie/treemap)
}

interface Layout {
  slots: Slot[];
  assignment: Map<number, number>;  // slotId → colorIndex
}
```

**Slot definitions by viz type:**

| Viz Type | Slots | Adjacency | Importance |
|:---------|:------|:----------|:-----------|
| **Pie Chart** | N segments | Circular (each has 2 neighbors) | 12 o'clock = highest |
| **Bar Chart** | N bars | Linear (ends have 1, rest have 2) | Leftmost = highest |
| **Treemap** | N rectangles | Variable (shared edges) | Top-left = highest |

#### Objective Function

We maximize a score that rewards:
1. **High perceptual distance between neighbors** (distinguishability)
2. **High contrast between neighbors** (visual separation)
3. **Prominent colors in important slots** (visual hierarchy)

```typescript
function evaluateLayout(
  assignment: Map<number, number>,
  slots: Slot[],
  distMatrix: number[][],
  contrastMatrix: number[][],
  prominence: number[]
): number {
  let score = 0;

  // Adjacency score: reward high distance and contrast between neighbors
  for (const slot of slots) {
    const colorA = assignment.get(slot.id)!;
    for (const neighborId of slot.neighbors) {
      const colorB = assignment.get(neighborId)!;
      if (colorA < colorB) {  // count each pair once
        // Normalize: deltaE typically 0-100, contrast 1-21
        const distScore = distMatrix[colorA][colorB] / 100;
        const contrastScore = (contrastMatrix[colorA][colorB] - 1) / 20;
        score += (distScore * 0.5 + contrastScore * 0.5);
      }
    }
  }

  // Placement score: prominent colors in important slots
  for (const slot of slots) {
    const colorIdx = assignment.get(slot.id)!;
    score += prominence[colorIdx] * slot.importance * 0.3;
  }

  return score;
}
```

#### Optimization Algorithm

For typical palette sizes (4-8 colors), we use **brute force with fallback**:

```typescript
function optimizeLayout(
  colors: PaletteColor[],
  slots: Slot[]
): Map<number, number> {
  const n = colors.length;
  const distMatrix = buildDistanceMatrix(colors);
  const contrastMatrix = buildContrastMatrix(colors);
  const prominence = colors.map(c => calculateProminence(c));

  if (n <= 6) {
    // Brute force: try all n! permutations (max 720)
    return bruteForceOptimal(slots, distMatrix, contrastMatrix, prominence);
  } else {
    // Greedy + local search for larger palettes
    return greedyWithSwapOptimization(slots, distMatrix, contrastMatrix, prominence);
  }
}

function bruteForceOptimal(
  slots: Slot[],
  distMatrix: number[][],
  contrastMatrix: number[][],
  prominence: number[]
): Map<number, number> {
  const n = prominence.length;
  let bestAssignment: Map<number, number> | null = null;
  let bestScore = -Infinity;

  // Generate all permutations
  for (const perm of permutations(range(n))) {
    const assignment = new Map(slots.map((s, i) => [s.id, perm[i]]));
    const score = evaluateLayout(assignment, slots, distMatrix, contrastMatrix, prominence);
    if (score > bestScore) {
      bestScore = score;
      bestAssignment = assignment;
    }
  }

  return bestAssignment!;
}

function greedyWithSwapOptimization(
  slots: Slot[],
  distMatrix: number[][],
  contrastMatrix: number[][],
  prominence: number[]
): Map<number, number> {
  // Start: assign by prominence (highest prominence → highest importance slot)
  const sortedSlots = [...slots].sort((a, b) => b.importance - a.importance);
  const sortedColors = prominence.map((p, i) => ({ p, i })).sort((a, b) => b.p - a.p);
  const assignment = new Map(sortedSlots.map((s, i) => [s.id, sortedColors[i].i]));

  // Local search: keep swapping until no improvement
  let improved = true;
  while (improved) {
    improved = false;
    const currentScore = evaluateLayout(assignment, slots, distMatrix, contrastMatrix, prominence);

    for (let i = 0; i < slots.length; i++) {
      for (let j = i + 1; j < slots.length; j++) {
        // Try swap
        const slotA = slots[i].id, slotB = slots[j].id;
        const colorA = assignment.get(slotA)!, colorB = assignment.get(slotB)!;
        assignment.set(slotA, colorB);
        assignment.set(slotB, colorA);

        const newScore = evaluateLayout(assignment, slots, distMatrix, contrastMatrix, prominence);
        if (newScore > currentScore) {
          improved = true;
          break;  // restart search
        } else {
          // revert
          assignment.set(slotA, colorA);
          assignment.set(slotB, colorB);
        }
      }
      if (improved) break;
    }
  }

  return assignment;
}
```

### Mockup Slides Algorithm

The Mockup slides (UI Components, Dashboard, Mobile App) use a **semantic slot tree** algorithm that assigns palette colors to UI elements based on their hierarchical relationships and semantic roles.

#### Key Differences from Data Viz

| Aspect | Data Viz | Mockups |
|:-------|:---------|:--------|
| Structure | Flat slots (pie slices, bars) | Tree hierarchy (containers → children) |
| Relationships | Neighbors (side-by-side) | Parent-child + Siblings |
| Constraints | Soft optimization | Hard contrast requirements |
| Color source | Palette colors only | Palette + derived scales |

#### Tree-Based Slot Structure

UI mockups are modeled as a tree where each node represents a semantic UI element:

```
page-bg (root)
├── header-bg
│   └── header-text
├── card-bg
│   ├── badge-1 ─┐
│   ├── badge-2 ─┼── siblings
│   └── badge-3 ─┘
└── btn-primary-bg
    └── btn-primary-text
```

#### Relationships

**Parent-Child** (containment): Child elements sit ON their parent. They require **contrast** for visibility.

```
┌─────────────────────┐
│  Button Background  │  ← parent
│    "Click me"       │  ← child (text needs contrast with bg)
└─────────────────────┘
```

**Siblings** (same container): Elements within the same parent that need to be distinguished. They require **perceptual distance**.

```
┌─────────────────────────────────┐
│  Card Background (parent)       │
│                                 │
│  [Badge 1] [Badge 2] [Badge 3]  │  ← siblings (need distance from each other)
│                                 │     + all need contrast with card-bg
└─────────────────────────────────┘
```

#### Slot Definition

```typescript
interface SlotNode {
  id: string;
  type: 'background' | 'surface' | 'accent' | 'text' | 'border';
  importance: number;       // 0-1, for prominence matching
  minContrast?: number;     // WCAG ratio required with parent (e.g., 4.5 for text)
  children: SlotNode[];     // child slots (optimized after this node gets its color)
}
```

| Property | Description |
|:---------|:------------|
| `id` | Unique identifier for this slot |
| `type` | Semantic type - determines which part of color scale to use |
| `importance` | How prominent this element should be (affects color prominence matching) |
| `minContrast` | Minimum WCAG contrast ratio with parent (null for root) |
| `children` | Child slots that will be optimized once this slot's color is assigned |

#### Color Scale Generation

When the palette has fewer colors than slots, we generate derived colors using scales:

```typescript
// For each palette color, generate a 9-step scale (white → color → black)
function generateScale(hex: string): string[] {
  const lightScale = chroma.scale(['#ffffff', hex]).mode('lab').colors(5);
  const darkScale = chroma.scale([hex, '#000000']).mode('lab').colors(5);
  // Combine: [0-3] light, [4] original, [5-8] dark
  return [...lightScale.slice(0, 4), hex, ...darkScale.slice(1)];
}

// Example: #4a9eff (blue) →
// [#ffffff, #c4dfff, #8fc7ff, #6ab0ff, #4a9eff, #2a7edf, #1a5ebf, #0a3e9f, #000000]
//  pos 0    pos 1    pos 2    pos 3    pos 4    pos 5    pos 6    pos 7    pos 8
```

#### Expanded Color Pool

The color pool includes scales from all palette colors plus a neutral gray scale:

```typescript
function expandPalette(colors: PaletteColor[]): ExpandedColor[] {
  const pool: ExpandedColor[] = [];

  // Add palette colors with their scales
  for (let i = 0; i < colors.length; i++) {
    const scale = generateScale(colors[i].hex);
    for (let j = 0; j < scale.length; j++) {
      pool.push({
        hex: scale[j],
        sourceIndex: i,
        scalePosition: j,
        prominence: calculateProminence(scale[j]),
      });
    }
  }

  // Add neutral gray scale for text contrast flexibility
  const neutralGrays = chroma.scale(['#ffffff', '#000000']).mode('lab').colors(9);
  for (let j = 0; j < neutralGrays.length; j++) {
    pool.push({
      hex: neutralGrays[j],
      sourceIndex: -1, // neutral marker
      scalePosition: j,
      prominence: calculateProminence(neutralGrays[j]),
    });
  }

  return pool;
}
```

#### Slot Types and Scale Selection

Different slot types pick from different parts of the color scale:

| Slot Type | Scale Region | Use Case |
|:----------|:-------------|:---------|
| `background` | Darkest (6-8) | Page backgrounds, containers |
| `surface` | Dark (5-7) | Cards, panels, elevated surfaces |
| `accent` | Mid-range (2-5) | Primary colors, CTAs, highlights |
| `text` | Light (0-2) OR Dark (6-8) | Text - both candidates, contrast decides |
| `border` | Muted mid-tones (3-5) | Subtle dividers, input borders |

```typescript
function getScaleRegion(type: SlotType): [number, number] {
  switch (type) {
    case 'background': return [6, 8]; // darkest
    case 'surface':    return [5, 7]; // dark
    case 'accent':     return [2, 5]; // includes original (4)
    case 'text':       return [0, 2]; // lightest (dark also checked via filter)
    case 'border':     return [3, 5]; // muted mid-tones
  }
}
```

#### Text Color Selection

For text slots, both light and dark candidates are considered. The algorithm picks the color with best contrast against the parent background:

```typescript
function getTextColor(
  bgHex: string,
  assignments: Map<string, string>,
  textSlotId: string,
  colorPool?: ExpandedColor[]
): string {
  // Use assigned color if available
  const assigned = assignments.get(textSlotId);
  if (assigned) return assigned;

  // Find best contrast from pool candidates (positions 0-2 and 6-8)
  if (colorPool && colorPool.length > 0) {
    const textCandidates = colorPool.filter(
      (c) => c.scalePosition <= 2 || c.scalePosition >= 6
    );

    let bestColor = '#000000';
    let bestContrast = 0;

    for (const candidate of textCandidates) {
      const contrastVal = chroma.contrast(bgHex, candidate.hex);
      if (contrastVal > bestContrast) {
        bestContrast = contrastVal;
        bestColor = candidate.hex;
      }
    }
    return bestColor;
  }

  // Fallback to black/white
  return chroma.contrast(bgHex, '#ffffff') > chroma.contrast(bgHex, '#000000')
    ? '#ffffff' : '#000000';
}
```

#### Top-Down Optimization

The algorithm processes the tree level by level, from root to leaves. **Important:** Children are processed as sibling groups using `optimizeSiblings`, not by recursive `optimizeTree` calls (which would re-assign parent colors):

```typescript
function optimizeTree(
  node: SlotNode,
  colorPool: ExpandedColor[],
  parentColor: string | null = null
): Map<string, string> {
  const assignments = new Map<string, string>();

  // 1. Assign this node
  const nodeAssignment = optimizeSiblings([node], colorPool, parentColor);
  const nodeColor = nodeAssignment.get(node.id)!;
  assignments.set(node.id, nodeColor);

  // 2. Optimize children as siblings (they share this node as parent)
  if (node.children.length > 0) {
    const childAssignments = optimizeSiblings(node.children, colorPool, nodeColor);
    for (const [id, hex] of childAssignments) {
      assignments.set(id, hex);
    }

    // 3. Process grandchildren directly (don't re-assign the child)
    for (const child of node.children) {
      if (child.children.length > 0) {
        const childColor = assignments.get(child.id)!;
        const grandchildAssignments = optimizeSiblings(
          child.children,
          colorPool,
          childColor
        );
        for (const [id, hex] of grandchildAssignments) {
          assignments.set(id, hex);
        }

        // Continue for great-grandchildren if needed
        for (const grandchild of child.children) {
          if (grandchild.children.length > 0) {
            const grandchildColor = assignments.get(grandchild.id)!;
            const greatGrandchildAssignments = optimizeSiblings(
              grandchild.children,
              colorPool,
              grandchildColor
            );
            for (const [id, hex] of greatGrandchildAssignments) {
              assignments.set(id, hex);
            }
          }
        }
      }
    }
  }

  return assignments;
}
```

#### Sibling Optimization

When optimizing siblings (elements at the same level with the same parent):

```typescript
function optimizeSiblings(
  siblings: SlotNode[],
  colorPool: ExpandedColor[],
  parentColor: string | null
): Map<string, string> {
  // 1. Filter colors by type (each sibling picks from its type's scale region)
  const candidatesPerSlot = siblings.map(s =>
    colorPool.filter(c => isInScaleRegion(c, s.type))
  );

  // 2. If parent exists, further filter by contrast requirement
  if (parentColor) {
    for (let i = 0; i < siblings.length; i++) {
      if (siblings[i].minContrast) {
        candidatesPerSlot[i] = candidatesPerSlot[i].filter(c =>
          chroma.contrast(c.hex, parentColor) >= siblings[i].minContrast!
        );
      }
    }
  }

  // 3. Optimize assignment to maximize:
  //    - Perceptual distance between siblings (CIEDE2000)
  //    - Prominence matching (important slots get prominent colors)
  return bruteForceOrGreedy(siblings, candidatesPerSlot);
}
```

#### Objective Function

Similar to Data Viz, but with hard contrast constraints:

```typescript
function evaluateSiblingAssignment(
  assignment: Map<string, ExpandedColor>,
  siblings: SlotNode[],
  parentColor: string | null
): number {
  let score = 0;

  // 1. Contrast satisfaction (HARD constraint - already filtered, but penalize low margin)
  if (parentColor) {
    for (const node of siblings) {
      const color = assignment.get(node.id)!;
      const contrast = chroma.contrast(color.hex, parentColor);
      const margin = contrast - (node.minContrast || 1);
      score += Math.min(margin / 10, 0.5);  // bonus for exceeding requirement
    }
  }

  // 2. Sibling distance (perceptual distinctness)
  for (let i = 0; i < siblings.length; i++) {
    for (let j = i + 1; j < siblings.length; j++) {
      const colorA = assignment.get(siblings[i].id)!;
      const colorB = assignment.get(siblings[j].id)!;
      const dist = chroma.deltaE(colorA.hex, colorB.hex);
      score += Math.min(dist / 100, 1);  // normalized 0-1
    }
  }

  // 3. Prominence matching
  for (const node of siblings) {
    const color = assignment.get(node.id)!;
    score += color.prominence * node.importance * 0.3;
  }

  return score;
}
```

#### Optimization Flow

1. **Assign page-bg**: Pick darkest shade from palette
2. **Optimize level 1 siblings** (header-bg, stat-1, stat-2, stat-3, chart-area):
   - All must contrast with page-bg (≥3:1 for accents, ≥1.5:1 for surfaces)
   - Maximize perceptual distance between stat cards
   - Match importance to prominence
3. **For each level 1 node, optimize its children**:
   - header-text must contrast with header-bg (≥4.5:1)
   - stat-N-text must contrast with stat-N (≥4.5:1)
   - bar-N must contrast with chart-area and be distant from other bars

**OkHSL Wheel**:
- **Circular hue**: 0-360° around the wheel (red→yellow→green→cyan→blue→magenta)
- **Radial saturation**: Center = gray (0%), Edge = fully saturated (100%)
- **All palette colors as markers**: Visible on the wheel showing their hue/saturation position
- **Draggable markers**: Drag any marker to change its hue/saturation
- **Selected marker**: Larger, with luminance control visible

**Lightness Scrubber**:
- **Continuous slider**: Draggable handle for precise lightness control (0-100%)
- **Only for selected color**: Other markers don't show lightness (just position)
- **Visual feedback**: Track shows gradient from dark to light

**Color Information Panel**:
- **Preview swatch**: Current color with name
- **Hex code**: Current color value
- **Psychology tags**: Color associations (Warmth, Comfort, Earthiness, etc.)

**Interaction Model**:
1. Click a color strip → Picker opens with that color selected
2. Drag any marker on wheel → Changes that color's hue/saturation
3. Click a different marker → Selects it (shows its lightness scrubber)
4. Drag lightness handle → Changes selected color's lightness
5. Close picker → Changes are applied to palette

**OkHSL Benefits**:
- Perceptually uniform: equal steps look equal to human eye
- Intuitive cylindrical model: hue wheel is familiar to designers
- All palette colors visible: see relationships at a glance
- Predictable editing: drag to adjust without color shifts

- `set_palette` - Palette state changes
- `execute_js` - JavaScript executed in page context

**Architecture**:
- **Python MCP** handles SQLite persistence (insert, query, delete)
- **TypeScript** is read-only (queries via MCP, displays in UI)

### AI Generation Usage Flow

1. User adds colors to palette and locks one as the base
2. User asks: "Generate warm, earthy tones based on my locked green"
3. Claude calls `get_generation_request` to see locked colors and unlocked slot IDs
4. Claude generates harmonious colors based on the request
5. Claude calls `set_generated_colors` with colors mapped to slot IDs
6. Browser updates only the unlocked slots, preserving locked colors

**Important**: The `set_generated_colors` tool only accepts colors for unlocked slots. Locked colors cannot be modified through this tool - they serve as anchors for generation.

### File I/O Usage Flow

1. User asks: "Save this palette for later"
2. Claude calls `export_palette` with a path
3. User asks: "Show me a styled palette sheet"
4. Claude calls `generate_palette_sheet` with `output='browser'`
5. User asks: "Export this as a PDF for the client"
6. Claude calls `generate_palette_sheet` with `output='pdf'` and path
7. User asks: "Load my saved palette"
8. Claude calls `load_palette` with the file path

### Page Integration Usage Flow

1. User asks: "Apply my palette to this page"
2. Claude calls `get_page_structure` to understand the DOM skeleton
3. Claude analyzes the structure and identifies key selectors (`.site-header`, `.cta-button`, etc.)
4. Claude writes CSS using palette variables: `.cta-button { background: var(--primary) }`
5. Claude calls `inject_css` to apply the styles
6. Action is automatically recorded to `.xray/data.db`

**CSS Variables:** Webtools automatically injects palette colors as CSS variables when palette state changes:
- Colors with roles → `--{role}` (e.g., `--primary`, `--accent`)
- Colors without roles → `--color-1`, `--color-2`, etc.

### Action History Usage Flow

1. Claude makes modifications: CSS injections, palette changes, DOM edits
2. Each modification is automatically recorded to `.xray/data.db`
3. User navigates to a different page
4. User asks: "Apply the same color changes here"
5. Claude calls `get_action_history` to query relevant actions
6. Claude decides which modifications apply and adapts them for the new page

## Implementation Notes

### Color Generation Algorithms

Use **chroma.js** for color space manipulation. All algorithms work in HSL space with hue rotation:

| Algorithm          | Formula                                          |
|:-------------------|:-------------------------------------------------|
| Complementary      | `hue + 180°` (spread ±30° when multiple colors)  |
| Analogous          | `hue ± 30°` (spread ±45° when 3+ colors)         |
| Triadic            | `hue + 120°`, `hue + 240°` (spread 100°-260° when 3+ colors) |
| Split-Complementary| `hue + 150°`, `hue + 210°` (spread 130°-230° when 3+ colors) |
| Tetradic           | `hue + 90°`, `hue + 180°`, `hue + 270°` (spread 60°-300° when 4+ colors) |
| Monochromatic      | Same hue, vary saturation/lightness              |

**Locked colors behavior**: Use the first locked color as the base hue. If no colors are locked, use a random base.

**Complementary with multiple colors**: By definition, complementary colors are at 180° from each other, providing only 2 harmonious positions. When generating more than 1 unlocked color, the algorithm spreads them across a ±30° range centered on the complement (150° to 210°). This maintains the "opposite side of the wheel" identity while giving each color a distinct hue:

| Count | Offsets                    |
|:------|:---------------------------|
| 1     | 180°                       |
| 2     | 150°, 210°                 |
| 3     | 150°, 180°, 210°           |
| 4     | 150°, 170°, 190°, 210°     |

**Analogous with multiple colors**: Analogous colors are adjacent on the color wheel (±30° from base). When generating 3+ unlocked colors, the range widens to ±45° to give each color more breathing room. For odd counts, the distribution shifts by 5° to avoid generating a color at 0° (same hue as the locked base):

| Count | Offsets                        |
|:------|:-------------------------------|
| 1     | -30°                           |
| 2     | -30°, 30°                      |
| 3     | -17.5°, 5°, 27.5° (shifted)    |
| 4     | -27°, -9°, 9°, 27°             |

**Triadic with multiple colors**: Triadic colors are at 120° intervals, giving only 2 positions besides the base (120° and 240°). When generating 3+ unlocked colors, the algorithm spreads them evenly across the 100°-260° range, which encompasses both triadic points:

| Count | Offsets                        |
|:------|:-------------------------------|
| 1     | 120°                           |
| 2     | 120°, 240°                     |
| 3     | 100°, 180°, 260°               |
| 4     | 100°, 153°, 207°, 260°         |

**Split-Complementary with multiple colors**: Split-complementary flanks the complement with two colors at 150° and 210° (only 60° apart). When generating 3+ unlocked colors, this narrow range causes clustering. The algorithm widens to 130°-230° (100° range) to give each color space:

| Count | Offsets                        |
|:------|:-------------------------------|
| 1     | 150°                           |
| 2     | 150°, 210°                     |
| 3     | 130°, 180°, 230°               |
| 4     | 130°, 163°, 197°, 230°         |

**Tetradic with multiple colors**: Tetradic forms a rectangle on the color wheel with 4 positions total (base, 90°, 180°, 270°). This provides 3 unlocked positions. When generating 4+ unlocked colors (5+ total), the algorithm spreads them across the 60°-300° range:

| Count | Offsets                        |
|:------|:-------------------------------|
| 1     | 90°                            |
| 2     | 90°, 180°                      |
| 3     | 90°, 180°, 270°                |
| 4     | 60°, 140°, 220°, 300°          |
| 5     | 60°, 120°, 180°, 240°, 300°    |

    TOOLS = [
        "get_generation_request",
        "set_generated_colors",
        "load_palette",
        "export_palette",
        "generate_palette_sheet",
    ]

    async def _handle_generate_palette_sheet(self, arguments: dict) -> list[TextContent]:
        """Generate palette sheet - display in browser or save as PDF."""
        output = arguments.get("output")  # "browser" or "pdf"
        sheet = arguments.get("sheet")
        path = arguments.get("path")

        # Get palette state from browser
        palette_data = await self._get_palette_state()

        # Prepare template context (merge palette state with enrichments)
        context, warnings = self._prepare_sheet_context(sheet, palette_data)
        html = render_palette_report(context)

        if output == "browser":
            # Display in a new browser tab
            page = await self.browser.page.context.new_page()
            await page.set_content(html, wait_until="networkidle")
            await page.bring_to_front()
            return self._text(json.dumps({"status": "success", "warnings": warnings}))
        else:
            # Generate PDF and open it in the same tab (no flicker)
            page = await self._generate_pdf(html, Path(path))
            await page.goto(f"file://{Path(path).resolve()}")
            await page.bring_to_front()
            return self._text(json.dumps({
                "status": "success",
                "path": str(path),
                "warnings": warnings
            }))

    async def _generate_pdf(self, html: str, path: Path) -> Page:
        """Generate PDF from HTML using existing browser context."""
        page = await self.browser.page.context.new_page()
        await page.set_content(html, wait_until="networkidle")
        await page.emulate_media(media="screen")  # Preserve colors
        await page.pdf(path=str(path), format="A4", print_background=True)
        return page  # Return page so caller can navigate to PDF

        const rgb = oklchToRgb(L, C, this.hue);
        ctx.fillStyle = rgb;
        ctx.fillRect(x, y, 1, 1);
      }
    }
  }

## AI Theming Analysis

This section captures learnings from a 10-iteration AI-assisted theming session. See [iteration-notes](./004-color-palette-panel-iteration-notes.md) for detailed per-iteration breakdown.

### What Worked Well

1. **Structure over color extraction** - The pivot from `analyze_page_colors` to `get_page_structure` was correct. AI needs to understand *what* elements exist (DOM skeleton with semantic tags, IDs, classes) rather than just extracting existing colors.

2. **Reactive CSS variables** - Moving injection to webtools instead of an MCP handler was the right call. Palette state changes automatically sync to `--primary`, `--accent`, `--color-1` etc. without extra tool calls.

3. **Computed colors + a11y flags** - Adding `data-bg`, `data-fg`, and `data-a11y` attributes to `get_page_structure` output helps AI understand current styling and accessibility requirements.

4. **The iteration loop** - Human feedback at each step was invaluable. The qualitative scoring (0-10) forced the AI to calibrate improvements.

### What Could Improve

1. **CSS accumulation problem** - Each `inject_css` call adds rules on top of previous ones. By iteration 9, conflicts broke the table styling. A single CSS file with diff-based edits would be better.

2. **No rollback mechanism** - When an iteration breaks things, there's no easy way to revert. Versioned CSS states would help.

3. **Context window pressure** - 10 iterations with screenshots consumed context quickly. Structured approach (CSS file + targeted screenshots of affected areas) would be more sustainable.

4. **A11y flags underutilized** - The `data-a11y="contrast-sensitive"` flag warned about nav contrast, but the AI ignored it and made the nav unreadable anyway. Tools are only useful if the AI actually respects them.

### Stress Test Revelations

When the user intentionally held back guidance to stress-test the tools:

1. **Without motivation, AI flails** - Random "make it better" iterations felt disconnected. Choosing "Aurora Borealis" as a theme motivation (iteration 6) unified everything.

2. **Content analysis skipped** - AI didn't recognize it was a documentation site and applied vibrant gradients. The `get_page_structure` tool provides content hints, but AI didn't use them well.

3. **Same mistakes repeated** - Breaking elements, ignoring contrast, accumulating CSS. A disciplined checklist would help.

4. **Scoring required for calibration** - Without 0-10 feedback, AI had no sense of whether changes were improvements.

### Recommendations for Tool Improvements

#### MCP Context Enhancements

1. **Add content type hints** - `get_page_structure` should include a `content_type` field (e.g., "documentation", "marketing", "dashboard") inferred from page structure.

2. **Forced theme selection** - Before any styling, require the user to choose or confirm a "design motivation" (minimal, playful, corporate, thematic).

3. **A11y enforcement** - When `data-a11y="contrast-sensitive"` is present, the tool should validate proposed CSS changes against WCAG contrast ratios before applying.

#### CSS State Management

1. **Use a CSS file** - Instead of `inject_css` accumulation, maintain a single `/tmp/theme.css` file and use diff-based edits.

2. **Structured sections** - Organize CSS by component:
   ```css
   /* === VARIABLES === */
   /* === HEADER/NAV === */
   /* === MAIN CONTENT === */
   /* === FOOTER === */
   ```

3. **Reset before restyle** - When changing a component, first reset it with `all: revert` to avoid conflicts.

4. **Version snapshots** - Before each iteration, save CSS state so rollback is possible.

#### Process Improvements

1. **Pre-flight analysis** - Before styling, AI should:
   - Analyze screenshot content to understand page purpose
   - Check a11y flags and note constraints
   - Propose a theme motivation for user approval

2. **Post-iteration checklist** - After each change, verify:
   - [ ] Header readable?
   - [ ] Tables intact?
   - [ ] Links working?
   - [ ] No broken layouts?

3. **Targeted feedback** - Instead of full-page screenshots, capture specific regions that changed.
